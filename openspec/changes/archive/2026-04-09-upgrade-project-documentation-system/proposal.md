## Why

当前项目已经有 `README.md` 和 `docs/` 目录，但内容主要停留在功能概览与分散的模块说明，缺少能够帮助维护者快速建立系统模型的完整文档体系。随着功能和实现细节增加，项目需要一套更清晰的“入口文档 + 架构文档 + 调试排障文档 + 维护约束”结构，来系统说明真实运行工作流、数据流转、模块边界以及定位问题的方法。

## What Changes

- 重构仓库文档信息架构，明确 `README.md` 与 `docs/` 的职责边界，形成面向不同读者层级的导航体系
- 增补系统级文档，详细描述从应用启动、权限检查、快捷键监听、录音识别、LLM 润色到文本注入的完整运行工作流
- 补充模块级文档，说明各核心组件的职责、依赖关系、关键状态、回调连接点和协作方式
- 补充数据与控制流文档，说明音频、识别文本、配置、日志与系统事件在各模块之间的流转路径
- 补充问题定位与调试文档，基于现有日志模块、权限依赖和系统集成点建立可执行的排障路径
- 为文档维护建立约束，确保后续功能变更时 README 与 `docs/` 可以同步演进，而不是再次失真

## Capabilities

### New Capabilities
- `documentation-system`: 为项目提供分层、可导航、可维护的工程文档体系，覆盖系统工作流、数据流、模块职责、调试排障与文档维护规则

### Modified Capabilities

## Impact

- `README.md`: 需要重写为项目入口文档，聚焦产品概览、快速上手、核心工作流摘要和文档导航
- `docs/README.md`: 需要升级为文档总索引，清晰组织架构、开发、调试和参考文档
- `docs/*.md`: 需要扩充或重组为更细粒度、更贴近源码实现的系统说明文档
- `Sources/VoiceInput/*.swift`: 作为文档事实来源，需要被系统梳理其运行链路、模块边界、状态切换与日志信号
- OpenSpec: 需要为文档体系建立明确需求，约束后续文档升级的覆盖范围与质量标准
