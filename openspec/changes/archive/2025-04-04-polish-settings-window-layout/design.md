## Context

当前设置窗口已经承担通用配置入口，但其布局在固定窗口尺寸下没有很好地约束垂直空间，导致首个 section 距顶部过远、底部操作按钮可见区域不足。问题集中在 `SettingsWindow.swift` 的现有几何参数和 section 堆叠方式，不涉及数据模型、持久化逻辑或新依赖。另一个问题位于状态栏菜单构建逻辑，`Settings...` 菜单项带有前置图标，而相邻项为纯文本，属于轻量的一致性修正。

## Goals / Non-Goals

**Goals:**
- 在保留现有设置项结构的前提下，让 Settings 窗口更紧凑、平衡、完整可用
- 确保底部 `Test` / `Save` 按钮在默认窗口尺寸下完整可见
- 让状态栏菜单中的 `Settings...` 与其他一级菜单项保持一致的文本风格

**Non-Goals:**
- 不重构 Settings 窗口为新的导航结构或分页结构
- 不新增设置项，不改动设置持久化逻辑
- 不调整 `Language`、`Quit` 等其他菜单项行为或排序

## Decisions

### Decision: 在现有窗口结构上做参数级布局收紧
通过缩小顶部 inset、精简 section 间距、重新分配底部按钮区域高度来解决视觉与可用性问题，而不是重写整套 Settings 视图层次。

备选方案：
- 增加窗口整体高度。优点是实现简单；缺点是只是掩盖布局问题，窗口会显得更空。
- 引入滚动容器。优点是可承载更多设置；缺点是对当前轻量窗口来说过重，也会削弱简洁感。

选择现有方案，因为它能以最小改动修复当前两个核心问题，并保持窗口气质克制。

### Decision: 保持 section 结构不变，仅统一边距和底部动作节奏
保留 `General`、`Keyboard Shortcut`、`LLM Refinement` 的顺序和内容，避免对现有交互心智造成不必要影响。调整重点放在 section 标题与字段块之间的呼吸感，以及底部按钮与输入区之间的间隔。

备选方案：
- 合并 section 或折叠部分配置。这样会改变现有信息架构，超出本次范围。

### Decision: 直接移除 `Settings...` 菜单项的前置图标
菜单一致性问题属于呈现层问题，最小实现方式是保持菜单项文案和快捷键不变，仅使用纯文本标题。

备选方案：
- 给所有一级菜单项统一补图标。这样改动面更大，也不符合当前菜单的既有风格。

## Risks / Trade-offs

- [窗口过于紧凑] → 通过保留 section 分隔和稳定的控件间距，避免为了压缩高度而损失可读性
- [底部按钮位置调整影响不同字段状态] → 手动验证启用/禁用 LLM、长 API Key、焦点变化时的显示完整性
- [菜单项修改影响快捷键或点击目标] → 保持现有 action 和 key equivalent 不变，只调整标题展示

## Migration Plan

无需迁移数据或外部依赖。发布后仅需以手动方式验证：
1. 打开状态栏菜单，确认 `Settings...` 无前置图标
2. 打开 Settings，确认 `General` 更贴近顶部
3. 确认底部 `Test` / `Save` 按钮完整显示并可点击
